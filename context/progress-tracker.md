## Architecture Decisions

- Using shadcn/ui on Tailwind v4 with CSS-variable-based dark theme (dark only, no light mode).
- All color tokens defined as CSS custom properties in globals.css; shadcn semantic vars (--background, --primary, etc.) point to design system vars (--bg-base, --accent-primary, etc.).
- @theme inline maps design system tokens to Tailwind utilities: bg-surface, text-copy-primary, border-surface-border, text-brand, bg-brand-dim, etc.
- html element carries class="dark" so shadcn dark: variants always activate.
- Auth uses Clerk via `@clerk/nextjs` and `@clerk/ui`; route protection lives in `proxy.ts` at the project root (Next.js 16 file-convention rename of `middleware.ts`). Protected-by-default, with sign-in / sign-up paths derived from `NEXT_PUBLIC_CLERK_SIGN_IN_URL` / `NEXT_PUBLIC_CLERK_SIGN_UP_URL` exposed as public.
- `ClerkProvider` lives inside `<body>` in `app/layout.tsx` (required by `@clerk/nextjs` v7); appearance overrides reference the app's existing CSS variables — no hardcoded Clerk colors.
- Prisma 7 with the `prisma-client` generator outputting to `app/generated/prisma/` (gitignored). Schema is multi-file: top-level `prisma/schema.prisma` holds the generator + datasource, model files live under `prisma/models/`. Database access goes through the cached `lib/prisma.ts` singleton, which switches between Accelerate (`prisma+postgres://`) and the direct `@prisma/adapter-pg` driver based on the `DATABASE_URL` scheme. `prisma.config.ts` loads `.env` via `dotenv/config` and points `schema` at `prisma/`. The singleton is typed as `PrismaClient` (Accelerate-extended branch is cast through `unknown`) so model accessors stay strongly typed at call sites.
- API auth lives inside route handlers via `auth()` from `@clerk/nextjs/server`, returning JSON `401` when `userId` is null. `proxy.ts` short-circuits before `auth.protect()` for any `/api/*` path so API requests get the JSON `401` contract instead of Clerk's default `404` for non-document requests; document routes still go through `auth.protect()`. Mutation handlers fetch the target row, return `404` if missing and `403` if `ownerId !== userId` before applying the change.
- Editor sidebar data is fetched server-side. Each editor page (`(authenticated)/editor/page.tsx` and `(authenticated)/editor/[roomId]/page.tsx`) is an async server component that calls `getUserProjects()` from `lib/projects.ts` and passes `ownedProjects` / `sharedProjects` into `EditorShell`. The route-group layout `(authenticated)/layout.tsx` is a thin passthrough so the workspace shell mounts per-page (the layout cannot see `[roomId]` params, and the workspace navbar needs to show the current project's name). No client-side fetch for initial load — refresh comes from `router.refresh()` after a mutation, or `router.push()` after create/active-workspace delete.
- Project access checks happen in the page server component, not in middleware. `lib/project-access.ts` exposes `getCurrentIdentity()` (Clerk `auth()` + `currentUser()`) and `checkProjectAccess(projectId, identity)` (Prisma owner-or-collaborator match against any of the user's verified emails). The workspace page redirects unauthenticated users to `/sign-in`, and on access miss/deny mounts `<AccessDenied />` inside the `EditorShell` so the project sidebar stays available for navigation. The shell shows workspace-only chrome (project name in navbar, share button, AI sidebar + toggle) only when `currentProject` is provided.
- Collaborator identity is sourced from Clerk via the Backend API at request time. `ProjectCollaborator` rows store only the lowercased email; `lib/clerk-users.ts` enriches a row set on demand via `clerkClient().users.getUserList({ emailAddress })` and joins on lowercased email. There is no local user table — Clerk is the source of truth for display name and avatar, and the helper soft-fails to email-only when the lookup errors. Collaborator-list endpoints (`GET /api/projects/[projectId]/collaborators`) authorize via `checkProjectAccess` (owner OR collaborator); mutating endpoints (`POST` invite, `DELETE` remove) authorize via the project's `ownerId === userId` and return `403` for non-owners.
- Project ID = Liveblocks room ID. The create flow generates `${slug(name)}-${4-char-base36-suffix}` client-side and sends it as the optional `id` field on `POST /api/projects` so the Prisma `Project.id` (otherwise `cuid()`) is overridden with that value. URLs are `/editor/[projectId]` where `projectId` is that same string.
- Shared-projects query keys on Clerk email. `getUserProjects()` reads all `emailAddresses` from `currentUser()` and joins on `ProjectCollaborator.email IN (...)`, scoped to projects whose `ownerId !== userId` so an owner doesn't double-appear in the Shared tab.
- Liveblocks auth uses **access tokens** (`prepareSession` + `session.allow(roomId, FULL_ACCESS)`) rather than ID tokens. Project membership is enforced once at the auth route via `checkProjectAccess`, so rooms are created with `defaultAccesses: []` (private) and the access scope rides on the token itself — no per-user `usersAccesses` bookkeeping on the room. The Liveblocks node client is exposed as `getLiveblocksClient()` (lazy + cached on `globalThis`) instead of a module-level constant so the missing-env guard does not fire during `next build` page-data collection. `Liveblocks.UserMeta.info.avatar` is `string | undefined` (not `string | null`) because Liveblocks' core `IUserInfo` types `avatar` as `string`; the route omits the field entirely when Clerk has no `imageUrl`.
- Canvas state lives in Liveblocks Storage via `@liveblocks/react-flow`'s `useLiveblocksFlow({ suspense: true })`, which exposes a controlled-React-Flow surface (`nodes`, `edges`, `onNodesChange`, `onEdgesChange`, `onConnect`, `onDelete`) backed by `LiveMap`s under the storage key `"flow"`. The room is mounted client-side from `components/editor/canvas/canvas.tsx` (`LiveblocksProvider authEndpoint="/api/liveblocks-auth"` → `RoomProvider id={roomId} initialPresence={{ cursor: null, isThinking: false }}`), with `ClientSideSuspense` for first-load and `useErrorListener` for connection failures (codes -1/4001/4005/4006 each map to a tailored message). React Flow is configured with `ConnectionMode.Loose`, `fitView`, dot-pattern `<Background>` themed via CSS variables, and a `<MiniMap pannable zoomable>` styled for the dark theme. Custom node/edge rendering and persistence to Vercel Blob remain out of scope for this base.
- Canvas types are centralized in `types/canvas.ts`. Node data carries `{ label, color, shape }` keyed against the `NODE_COLORS` palette (8 fill/text pairs sourced from `ui-context.md`) and the `NODE_SHAPES` enumeration (`rectangle`/`diamond`/`circle`/`pill`/`cylinder`/`hexagon`). Custom node and edge type literals are pinned to `canvasNode` and `canvasEdge` so future custom renderers register against a single, shared discriminator.
- Drag-to-create flow uses HTML5 drag-and-drop with the custom MIME `application/x-ghost-shape` (constant `SHAPE_DRAG_MIME` in `components/editor/canvas/shape-panel.tsx`) carrying a `ShapeDragPayload` (`{ shape, width, height }`). `onDragOver` short-circuits unless `dataTransfer.types` includes that MIME, so unrelated drags (files, text) do not arm the canvas as a drop zone. Drop coordinates are converted via `useReactFlow().screenToFlowPosition({ clientX, clientY })` and offset by `width/2`/`height/2` so the drop point becomes the node's visual center. `useReactFlow()` requires an enclosing `<ReactFlowProvider>`; the inner flow is wrapped because the drop handler lives on the wrapper div, outside `<ReactFlow>`'s implicit provider scope. New nodes are added via the `useLiveblocksFlow` `onNodesChange` callback dispatching a `{ type: "add", item }` `NodeChange` (the hook does not expose a `setNodes`/`addNodes` shortcut).
- Node shape rendering is centralized in `components/editor/canvas/shape-render.tsx` (`<ShapeBackground>`). CSS-only shapes (`rectangle`, `pill`, `circle`) render as bordered absolutely-positioned divs; complex shapes (`diamond`, `hexagon`, `cylinder`) render as inline SVGs sized to the node's exact pixel bounds (`width`/`height` attributes with a matching `viewBox`, no `preserveAspectRatio` scaling) so stroke widths stay uniform regardless of aspect ratio. SVGs carry `overflow-visible` so polygon strokes near the viewBox edge are not clipped. `CanvasNode` selects between `var(--border-subtle)` at rest and `var(--accent-primary)` (with stroke width 2) when `selected`, and the label sits in an absolutely-positioned `pointer-events-none` overlay on top of the shape so it works for both CSS and SVG variants. The drag preview reuses the same `<ShapeBackground>` rendered in an `aria-hidden` off-screen container; `dataTransfer.setDragImage` snapshots the matching preview element at dragstart with a `(width/2, height/2)` offset so the cursor anchors at the shape's center.
- Per-node color selection uses xyflow's first-party `<NodeToolbar isVisible={selected} position={Position.Top}>` mounted from inside the custom node component, so the toolbar tracks the node's screen position automatically and only renders while the node is the active selection. The toolbar carries `nodrag nopan` plus `stopPropagation` on pointer/mouse/click events so swatch interactions never bubble up to React Flow's drag-or-pan gesture pipeline. Swatch clicks dispatch through `useReactFlow().updateNodeData(id, { color })`, which xyflow folds into a `replace` `NodeChange` and `useLiveblocksFlow` writes back to Storage — no separate API route, mutation, or local node-state copy. The active-color "ring" and the hover "glow" are both expressed in the swatch's paired `text` color (set inline as a `--swatch-glow` CSS variable consumed by a Tailwind arbitrary-value `group-hover:shadow-[0_0_6px_var(--swatch-glow)]`), so the toolbar visually echoes the same palette pairing the node will adopt.
- Node resizing uses xyflow's first-party `<NodeResizer>` mounted inside the custom node component with `isVisible={selected}`. NodeResizer dispatches `dimensions` `NodeChange`s via React Flow's controlled `onNodesChange`, which the Liveblocks flow hook writes to Storage as `width`/`height` updates — no separate state plumbing needed. Inline label editing is local-only state (`isEditing` in `CanvasNode`) toggled by `onDoubleClick`; the textarea is bound directly to `data.label` and writes back through `useReactFlow().updateNodeData(id, { label })`, which xyflow internally diffs into a `replace` `NodeChange` so updates flow through the same `onNodesChange` → `useLiveblocksFlow` pipeline as adds and resizes. The textarea uses Tailwind's `field-sizing-content` for content-driven height (so the static label and edit overlay occupy the same space) and carries `nodrag nopan nowheel` plus key/pointer/wheel `stopPropagation` so typing inside it never triggers canvas drag, pan, zoom, or delete shortcuts. `Escape` and `blur` close edit mode.
- Edges are rendered through the custom `canvasEdge` type registered in `EDGE_TYPES`. New connections inherit the type + `MarkerType.ArrowClosed` arrowhead via `<ReactFlow defaultEdgeOptions>` so `useLiveblocksFlow.onConnect` does not need to be wrapped to attach edge defaults. The custom edge calls `getSmoothStepPath` (`borderRadius: 8`) inside the renderer for right-angle routing and uses `<BaseEdge>`'s built-in `interactionWidth` (set to 24px) to widen the click/hover target without thickening the visible 1.5px stroke. Hover state is local-only (`onPointerEnter`/`onPointerLeave` on a wrapping `<g>`); selection is read from `EdgeProps.selected`; both feed an `isActive` flag that swaps stroke/opacity through CSS transitions — no extra props plumbing through Liveblocks. Edge connection points are four `Handle` components per node (`top`/`right`/`bottom`/`left`) all typed as `source`; with `ConnectionMode.Loose` already set globally, source-to-source connections are valid, so any handle can connect to any other handle. Handles render as 8x8 white circles with a `var(--bg-base)` border and fade in via `group-hover:opacity-100` on the node wrapper rather than imperative state.
- Edge label rendering uses xyflow's `<EdgeLabelRenderer>` portal positioned with the `[labelX, labelY]` tuple returned by `getSmoothStepPath` — never compute the midpoint manually. The label container is a `pointer-events: all` div with `nodrag nopan` plus `stopPropagation` on pointer/mouse/click/key events so dragging or typing inside the label never bubbles to React Flow's canvas drag/pan handlers. Saved labels render as a small rounded-full pill; an empty active edge (hovered or selected) renders a dashed faint "Add label" hint; an empty inactive edge renders nothing. Double-clicking the edge `<g>` or the label opens an `<input>` with `field-sizing-content` + `min-w-[8ch]` so it grows with typed content and stays clickable while empty. Edits write back via `useReactFlow<Node, CanvasEdge>().updateEdgeData(id, { label })`, which xyflow folds into a `replace` `EdgeChange` that flows through `useLiveblocksFlow.onEdgesChange` for collaborative sync — same pattern as the per-node color/label updates. `Enter`, `Escape`, and `blur` all commit (per spec) and close edit mode.
- Starter template imports clear the canvas via `useLiveblocksFlow`'s `onDelete({ nodes, edges })` callback, then dispatch pure `{ type: "add" }` change arrays through `onNodesChange` / `onEdgesChange`. The hook's `applyNodeChanges` / `applyEdgeChanges` explicitly no-op `{ type: "remove" }` entries (deletion is only persisted through the dedicated `onDelete` mutation), so a remove-then-add batch was silently dropping the removes and stacking the template on top of existing work. Template node/edge IDs are re-prefixed with `tpl-${Date.now()}-${counter}` at import time so repeated imports of the same template (or imports with literal IDs that already exist) never collide. The modal lives inside `CanvasFlowInner` (so it has access to `useLiveblocksFlow` and `useReactFlow`); `EditorShell` only owns the open boolean and exposes it through `<CanvasTemplatesProvider>`, which is mounted around the navbar + main so both the navbar trigger and the canvas modal share the same context. `fitView` runs in a `setTimeout(..., 80)` after the import dispatch so the controlled-flow rerender has committed the new node positions before xyflow computes the bounding box.
- Presence avatars and live cursors live inside the canvas room scope, never the shared editor navbar. The avatar group mounts as a `<Panel position="top-right">` so xyflow handles screen-space positioning (no fixed/absolute coords) and the panel only exists while the room is active. The Clerk `<UserButton>` is suppressed in the navbar (`hideUserButton={hasCurrentProject}` in `EditorShell`) so canvas view shows exactly one Clerk UserButton — inside the presence group — while the editor home navbar still renders the regular navbar UserButton. The `id` on `Liveblocks.UserMeta` matches Clerk's `userId` (the auth route uses `identity.userId` as the prepareSession ID), so presence rows are filtered with `info.id !== userId` from `useAuth()` to drop the current user from the collaborator list. Subscriptions use `useOthersMapped((other) => ({...}), shallow)` rather than `useOthers()` so the avatar list only re-renders on join/leave or info change (not on every cursor tick), and the cursors layer only re-renders when the projected `(id, name, color, cursor)` tuple changes for any user. Cursor positions are broadcast in **flow coordinates**, not screen coordinates, so cursors track the same logical canvas point across remote viewports with different pan/zoom; the broadcaster runs `screenToFlowPosition({ x: clientX, y: clientY })` then `updateMyPresence({ cursor: { x, y } })` on `<ReactFlow>`'s `onMouseMove` (the wrapper passes through DOM events) and clears with `cursor: null` on `onMouseLeave`. The renderer projects back via manual viewport math (`screenX = cursor.x * zoom + vx`, `screenY = cursor.y * zoom + vy` from `useViewport()`) and applies it as a `transform: translate(...)` on a `pointer-events-none absolute inset-0` overlay layered as a sibling of `<ReactFlow>` inside the wrapper `<div>` (which gained `relative`); cursors thus track the canvas without scaling with zoom. ViewportPortal was deliberately not used — it would scale cursors with zoom, requiring a counter-scale.
- The AI sidebar lives in `components/editor/ai-sidebar.tsx` as a self-contained client component; `EditorShell` only owns the `isAiOpen` boolean and forwards open/close. The sidebar surface itself is a pure UI shell — local state covers chat draft + `ChatMessage[]` and a starter-chip → draft handoff, but submit only appends the user message and does not call any backend, Liveblocks, or AI logic. The sidebar is conditionally mounted (`hasCurrentProject ? <AiSidebar /> : null`) so its state resets when the user switches between the editor home and a project workspace. Active tab styling overrides shadcn's default-variant `data-active:bg-background dark:data-active:bg-input/30` with `data-active:bg-brand-dim data-active:text-brand` (plus `dark:data-active:*` and `dark:data-active:border-transparent`) so the active tab tint matches the project sidebar's `bg-brand-dim`/`text-brand` accent — there is no separate "accent" Tailwind token in `globals.css`; brand-dim + brand is the project's accent pairing. Solid accent buttons (Send, Generate Spec) use `bg-brand text-white hover:bg-brand/90` rather than the shadcn `default` variant, because `--primary-foreground` is `var(--bg-base)` (dark) and the spec calls for white text on the cyan accent.
- Canvas autosave persists the live React Flow graph to Vercel Blob at `canvas/{projectId}.json` and stores the resulting blob URL on `Project.canvasJsonPath`. Persistence is one-way per writer: the autosave hook `PUT`s the latest in-memory `nodes`/`edges` after a 1500ms debounce; conflict resolution is already handled inside the Liveblocks room, so Vercel Blob is treated as a snapshot mirror, not a source of truth. The blob is uploaded `access: "private"` with `allowOverwrite: true` because the configured Vercel Blob store is private; reads must go through the Vercel Blob SDK's `get(url, { access: "private", useCache: false })` rather than a plain `fetch` (a public fetch on a private URL fails). Project access is gated server-side via `checkProjectAccess` at both write time (`PUT /api/projects/[projectId]/canvas`) and read time (the page's server-side `getSavedCanvas` call + the GET route). `lib/canvas-storage.ts` is the single read-side helper — both the page (server component) and the GET route delegate to it so blob fetching, defensive parsing, and the "no saved canvas yet" `null` case stay in one place. The autosave hook strips xyflow's transient per-render fields (`selected`, `measured`, `dragging`, etc.) before serializing, so cursor/selection churn never triggers a save. Loading reuses Liveblocks' `useLiveblocksFlow({ nodes: { initial }, edges: { initial } })` semantics rather than manually inspecting room emptiness — `initial` only seeds Storage when it is empty, so active collaboration on an existing room is preserved without any explicit check on the client. The hook pushes status + a `saveNow` callback up to `EditorShell` via a dedicated `CanvasSaveContext` rather than bubbling through the navbar prop chain, because the canvas surface lives several levels below the navbar (inside `RoomProvider` + `ReactFlowProvider`) and the React 19 `react-hooks/refs` rule blocks the simpler "lift refs up via props" alternative. The hook itself avoids the "queue another save inside the in-flight one's `.finally`" pattern because the React Compiler's `react-hooks/preserve-manual-memoization` rule rejects the resulting recursive `useCallback`; instead, post-in-flight saves rely on natural re-firing — the next change re-arms the debounce and `flush` runs again.
- Canvas viewport controls (zoom in/out/fit view) are driven by the React Flow instance returned by `useReactFlow()` — never by manual viewport math. Each handler passes a `{ duration }` option so transitions animate (200ms for zoom in/out, 300ms for fit view). Undo/redo are routed through Liveblocks history (`useUndo`/`useRedo`/`useCanUndo`/`useCanRedo` from `@liveblocks/react/suspense`) so they cover everything in Storage — node positions, sizes, labels, colors, edge labels — without any per-feature undo plumbing. The control bar reads `canUndo`/`canRedo` to dim disabled buttons; the keyboard shortcut hook re-uses the same booleans to no-op the matching shortcut. The shortcut hook lives in `hooks/use-keyboard-shortcuts.ts` (parallel to `hooks/use-project-actions.ts`); shortcuts are short-circuited when the active `event.target` is an `INPUT`, `TEXTAREA`, `SELECT`, or `contentEditable` element so typing in node/edge labels never accidentally zooms or undoes the canvas. The control bar mounts inside React Flow via `<Panel position="bottom-left">` (sibling of the existing `bottom-center` shape panel) so xyflow handles screen-space positioning automatically — no fixed/absolute coordinates.

## Session Notes

- Do not modify components/ui/* after installation per ai-workflow-rules.md.
- Design system palette is defined in globals.css :root; all other files reference tokens, never raw hex values.
- Keep Clerk's default `<UserButton>` and built-in profile flows intact — do not rebuild Clerk internals.
- React 19 lint rule `react-hooks/set-state-in-effect` blocks `setState` inside `useEffect`. For dialogs that need fresh state per open, mount/unmount conditionally from the parent rather than syncing props → state in an effect.
- Editor pages must remain server-rendered above `EditorShell` so they can do the initial server-side `getUserProjects()` (and `checkProjectAccess()`) fetch. Don't add `"use client"` to `(authenticated)/editor/page.tsx` or `(authenticated)/editor/[roomId]/page.tsx`. `EditorShell` itself is the client boundary.
- Prisma client output (`app/generated/prisma/`) is gitignored and recreated by `prisma generate`; never import directly from `@prisma/client` — always go through `lib/prisma.ts` or `@/app/generated/prisma/client`.
- `prisma migrate dev` only writes/applies migrations; if you need the generated client right after, run `prisma generate` (or rely on `next build` which triggers it via the generator).
- Per the React 19 `react-hooks/set-state-in-effect` rule, do not call `setState` synchronously inside a `useEffect` body. For dialogs whose state should reset on each open, mount/unmount the dialog conditionally from the parent and let initial state defaults handle the reset; state mutations from async handlers (then/catch) are fine.
- `useLiveblocksFlow` (`@liveblocks/react-flow`) silently ignores `{ type: "remove" }` entries in `onNodesChange` / `onEdgesChange`. Deletion only persists through the dedicated `onDelete({ nodes, edges })` callback. Any "clear and replace" flow must call `onDelete` first, then dispatch pure `{ type: "add" }` changes — building a combined remove+add batch will leave the existing graph in place.
- The Vercel Blob store is configured with private access. `put(...)` must use `access: "private"` (not `"public"`) and reads must go through the SDK's `get(url, { access: "private" })` rather than a public `fetch` on the blob URL.
- The React Compiler `react-hooks/preserve-manual-memoization` lint rule rejects recursive async `useCallback`s — calling the same callback from inside its own `.finally` (e.g. "if a save is queued, flush again") fails the rule. Replace recursive flushes with re-firing patterns (the next change re-runs the debounce effect, which calls flush again).
- The React 19 `react-hooks/refs` lint rule rejects ref mutation during render. To keep refs in sync with the latest props (e.g. so an async callback can read the latest `nodes`/`edges`), assign through a dedicated `useEffect` rather than writing `nodesRef.current = nodes` in the component body.
