## Architecture Decisions

- Using shadcn/ui on Tailwind v4 with CSS-variable-based dark theme (dark only, no light mode).
- All color tokens defined as CSS custom properties in globals.css; shadcn semantic vars (--background, --primary, etc.) point to design system vars (--bg-base, --accent-primary, etc.).
- @theme inline maps design system tokens to Tailwind utilities: bg-surface, text-copy-primary, border-surface-border, text-brand, bg-brand-dim, etc.
- html element carries class="dark" so shadcn dark: variants always activate.
- Auth uses Clerk via `@clerk/nextjs` and `@clerk/ui`; route protection lives in `proxy.ts` at the project root (Next.js 16 file-convention rename of `middleware.ts`). Protected-by-default, with sign-in / sign-up paths derived from `NEXT_PUBLIC_CLERK_SIGN_IN_URL` / `NEXT_PUBLIC_CLERK_SIGN_UP_URL` exposed as public.
- `ClerkProvider` lives inside `<body>` in `app/layout.tsx` (required by `@clerk/nextjs` v7); appearance overrides reference the app's existing CSS variables â€” no hardcoded Clerk colors.
- Prisma 7 with the `prisma-client` generator outputting to `app/generated/prisma/` (gitignored). Schema is multi-file: top-level `prisma/schema.prisma` holds the generator + datasource, model files live under `prisma/models/`. Database access goes through the cached `lib/prisma.ts` singleton, which switches between Accelerate (`prisma+postgres://`) and the direct `@prisma/adapter-pg` driver based on the `DATABASE_URL` scheme. `prisma.config.ts` loads `.env` via `dotenv/config` and points `schema` at `prisma/`. The singleton is typed as `PrismaClient` (Accelerate-extended branch is cast through `unknown`) so model accessors stay strongly typed at call sites.
- API auth lives inside route handlers via `auth()` from `@clerk/nextjs/server`, returning JSON `401` when `userId` is null. `proxy.ts` short-circuits before `auth.protect()` for any `/api/*` path so API requests get the JSON `401` contract instead of Clerk's default `404` for non-document requests; document routes still go through `auth.protect()`. Mutation handlers fetch the target row, return `404` if missing and `403` if `ownerId !== userId` before applying the change.
- Editor sidebar data is fetched server-side. `(authenticated)/layout.tsx` (async server component) calls `getUserProjects()` from `lib/projects.ts` and passes `ownedProjects` / `sharedProjects` into `EditorShell`. No client-side fetch for initial load â€” refresh comes from `router.refresh()` after a mutation, or `router.push()` after create/active-workspace delete.
- Project ID = Liveblocks room ID. The create flow generates `${slug(name)}-${4-char-base36-suffix}` client-side and sends it as the optional `id` field on `POST /api/projects` so the Prisma `Project.id` (otherwise `cuid()`) is overridden with that value. URLs are `/editor/[projectId]` where `projectId` is that same string.
- Shared-projects query keys on Clerk email. `getUserProjects()` reads all `emailAddresses` from `currentUser()` and joins on `ProjectCollaborator.email IN (...)`, scoped to projects whose `ownerId !== userId` so an owner doesn't double-appear in the Shared tab.
